### Security hardening

- **UFW firewall** active. Only these ports are open to the world:
  - `22/tcp` — SSH
  - `80/tcp` — HTTP (Traefik auto-redirects everything to HTTPS)
  - `443/tcp` — HTTPS (Traefik terminates TLS and routes by Host header)
  - Closed on the perimeter: `8000` (Coolify dashboard — now reached via `coolify.jouki.cz`), `6001-6002` (Coolify realtime — routed through `wss://coolify.jouki.cz/app`)
- **fail2ban** active with `sshd` jail
- **unattended-upgrades** enabled (OS security patches apply automatically)
- Kernel-upgrade reboot was pending at end of setup session — reboot at will, Docker containers auto-start via `restart: unless-stopped`

### Redeploy an app (after pushing new commit)

Replace `<APP_ID>` with `1` (backend) or `2` (landing). Runs on the server.

```bash
ssh root@178.104.160.182 '
cat > /tmp/rd.php << "PHPEOF"
$app = \App\Models\Application::find(<APP_ID>);
$uuid = \Illuminate\Support\Str::ulid()->toBase32();
queue_application_deployment(
    application: $app,
    deployment_uuid: $uuid,
    commit: "HEAD",
    force_rebuild: true,
    no_questions_asked: true,
);
echo "OK: " . $uuid . PHP_EOL;
exit;
PHPEOF
docker cp /tmp/rd.php coolify:/tmp/rd.php
docker exec -i coolify sh -c "php artisan tinker" < /tmp/rd.php 2>&1 | grep "OK:"
rm /tmp/rd.php
docker exec --user 0 coolify rm /tmp/rd.php 2>/dev/null
# Poll until finished
for i in 1 2 3 4 5 6; do
  S=$(docker exec coolify-db psql -U coolify -d coolify -tc "SELECT status FROM application_deployment_queues ORDER BY id DESC LIMIT 1;" | tr -d " ")
  echo "[$(date +%T)] $S"
  [ "$S" = "finished" ] || [ "$S" = "failed" ] && break
  sleep 15
done
'
```

### Update a field that the UI refuses to edit

Coolify's form validation occasionally reverts in-progress edits (Livewire + realtime sync race). Bypass via SQL:

```bash
ssh root@178.104.160.182 'docker exec coolify-db psql -U coolify -d coolify -c \
  "UPDATE applications SET fqdn=${quoted_value}, updated_at=NOW() WHERE id=1;"'
```

After editing, **always clear `custom_labels`** if you touched anything routing-related (`fqdn`, `redirect`, `is_force_https_enabled`), otherwise Coolify reuses the cached base64-encoded Traefik labels from the last deploy:

```sql
UPDATE applications SET custom_labels = NULL WHERE id = 1;
```

Then redeploy with `force_rebuild: true`.

### Decrypt a stored Postgres password (or any encrypted field)

Coolify stores credentials encrypted with the app's Laravel `APP_KEY`. Read them via artisan tinker:

```bash
ssh root@178.104.160.182 '
cat > /tmp/get.php << "PHPEOF"
$pg = \App\Models\StandalonePostgresql::find(1);
echo "User: " . $pg->postgres_user . PHP_EOL;
echo "Password: " . $pg->postgres_password . PHP_EOL;
echo "DB: " . $pg->postgres_db . PHP_EOL;
exit;
PHPEOF
docker cp /tmp/get.php coolify:/tmp/get.php
docker exec -i coolify sh -c "php artisan tinker" < /tmp/get.php 2>&1 | grep -E "^(User|Password|DB):"
rm /tmp/get.php
'
```

### Insert an environment variable (Coolify encrypts it)

Don't UPDATE the `environment_variables` table directly with plaintext — Coolify expects encrypted values. Use the model:

```php
$app = \App\Models\Application::find(1);
\App\Models\EnvironmentVariable::create([
    'key' => 'DATABASE_URL',
    'value' => 'postgres://...',
    'is_preview' => false,
    'is_build_time' => true,
    'is_runtime' => true,
    'is_literal' => false,
    'is_required' => false,
    'is_shared' => false,
    'resourceable_type' => \App\Models\Application::class,
    'resourceable_id' => $app->id,
]);
```

Note: Coolify's model observer automatically creates a second row with `is_preview=true` for every variable you create. That's expected — it's the preview-deploy copy.

## Gotchas we already hit

### 1. Form fields in Coolify UI silently revert during edit
Livewire + realtime sync pushes the server state back while you're typing. Fields in existing resources are especially prone to this (Postgres name, env vars, fqdn on a created application). **Workaround:** edit via SQL or artisan tinker as shown above.

### 2. `custom_labels` cache sticks after config change
Changing `applications.fqdn` in the DB without clearing `custom_labels` leaves the container launched with the stale Traefik labels (old hostname, wrong HTTPS router setup). Always `UPDATE applications SET custom_labels = NULL WHERE id = X` before the next deploy.

### 3. Coolify's injected healthcheck needs curl or wget
Regardless of the `HEALTHCHECK` you put in your Dockerfile, Coolify overrides it with `curl -f … || wget -q -O- …`. On `node:22-alpine` neither binary is present — the check fails immediately and the deploy is rolled back. **Always add `RUN apk add --no-cache curl`** to the runner stage.

### 4. `queue_application_deployment()` named param drift
Coolify's helper signature changes between beta releases. Expected-safe args:
```
queue_application_deployment(
    application: $app,
    deployment_uuid: $uuid,
    commit: "HEAD",
    force_rebuild: true,
    no_questions_asked: true,
);
```
If a name drifts (there was an `$is_new_deployment` that turned into something else), grep the current signature at `/var/www/html/bootstrap/helpers/applications.php:15` before calling.

### 5. Hetzner's "Coolify" pre-installed app image isn't actually Coolify
It ships Docker + a manual install script in `/opt/hcloud/coolify_setup.sh`. Run the script OR the upstream installer directly — don't expect Coolify to already be running after provisioning.

### 6. Opera vs Chrome extension compatibility
Our extension's `background.js` uses runtime feature detection (`typeof chrome.sidePanel !== 'undefined'`) — it's one folder loadable unpacked in both Chrome and Opera, no build step. This came up because Opera is Chromium-based but doesn't implement `chrome.sidePanel`. Unrelated to the server, but future you may get confused about why both browsers load the same `extension/` folder.
